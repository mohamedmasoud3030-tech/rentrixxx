## BUG-01 · Double Navigation After Login (High)

**File:** `src/hooks/use-auth.tsx` + `src/app/login-page.tsx`

**Problem:**  
When a user logs in, two separate navigation calls fire at the same time:

1. `use-auth.tsx` (line 88): `globalThis.location.replace('/')` — forces a full page reload
2. `login-page.tsx` useEffect: `supabase.auth.onAuthStateChange` fires `SIGNED_IN` → `router.navigate({ to: '/', replace: true })`

Both happen immediately after `signInWithEmail` resolves. This causes a race condition — the TanStack Router state gets reset mid-navigation by the hard `location.replace`, which can break router history and cause a flash/flicker on login.

**Fix:** Remove the `supabase.auth.onAuthStateChange` subscription from `LoginPage`. The `use-auth.tsx` listener already handles all session events globally. The `login-page.tsx` listener is redundant and harmful.

```diff
// login-page.tsx — remove this entire useEffect block:
- useEffect(() => {
-   const { data: { subscription } } = supabase.auth.onAuthStateChange((event, session) => {
-     if (event === 'SIGNED_IN' && session) {
-       router.navigate({ to: '/', replace: true });
-     }
-   });
-   return () => { subscription.unsubscribe(); };
- }, [router]);
```

## BUG-02 · Payment Mutation Retry=1 Breaks Idempotency (High)

**File:** `src/lib/query-client.ts`

**Problem:**  
The global `QueryClient` sets `mutations: { retry: 1 }`. The payment flow uses an atomic RPC (`record_invoice_payment_atomic`) with a `request_id` for idempotency. If the RPC call succeeds on the server but the response is lost (network drop), retrying with the **same** `request_id` is safe — the server deduplicates it.

However, if the retry fires with a **new** `request_id` (which can happen if the mutation component remounts), the payment gets recorded twice.

More critically: mutations that are NOT idempotent (e.g., `softDeleteContract`, expense creation) should never auto-retry.

**Fix:** Set `mutations: { retry: 0 }` globally. Add explicit retry logic only where safe.

```diff
// src/lib/query-client.ts
  mutations: {
-   retry: 1,
+   retry: 0,
  },
```

**Related:** `src/routes/_protected.accounting.tsx`

## BUG-06 · `receipts` Route Has No Dedicated Route File (Low)

**File:** `src/routeTree.ts` (line ~30)  
**Related:** All other routes have a file in `src/routes/`

**Problem:**  
Every other route imports its component from `src/routes/_protected.*.tsx`. The receipts route is the only exception — it imports directly from `src/features/financials/receipts/receipts-page.tsx`. There is no `_protected.receipts.tsx` file.

This is not a runtime bug, but it breaks the structural consistency of the codebase. If someone searches `src/routes/` to find the receipts route, they won't find it.

**Fix:** Create `src/routes/_protected.receipts.tsx` as a re-export (matching the pattern of every other route file), and update `routeTree.ts` to import from it.

## BUG-07 · `use-auth` Login Uses Hard Navigation (`globalThis.location.replace`) (Medium)

**File:** `src/hooks/use-auth.tsx` (line 88)

**Problem:**  
```ts
login: async (email, password) => {
  await signInWithEmail(email, password);
  globalThis.location.replace('/');  // ← hard page reload
},
```

`globalThis.location.replace('/')` triggers a full browser reload, which:
- Destroys TanStack Router state
- Destroys React Query cache (all prefetched data lost)
- Causes a visible flash before the app re-mounts

The same issue exists in `redirectToLogin()` using `globalThis.location.href`.

**Fix:** Inject a `router` reference into the auth context and use `router.navigate({ to: '/', replace: true })` instead. For `SIGNED_OUT`, the same — use router navigation.

## BUG-08 · `end_date >= start_date` Allows Same-Day Contracts (Low)

**File:** `src/features/contracts/contractSchema.ts`

**Problem:**  
```ts
.refine((value) => value.end_date >= value.start_date, ...)
```

This uses `>=`, which allows `start_date === end_date`. A zero-duration contract is logically invalid and will cause divide-by-zero or empty invoice generation in financial calculations.

**Fix:** Change to strict `>`:
```diff
- .refine((value) => value.end_date >= value.start_date, { ... })
+ .refine((value) => value.end_date > value.start_date, { ... })
```

Same fix applies to `renewalSchema`.
