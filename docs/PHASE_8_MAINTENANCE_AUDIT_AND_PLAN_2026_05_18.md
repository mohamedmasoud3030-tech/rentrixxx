# Phase 8 Maintenance Audit and Implementation Plan

Date: 2026-05-18
Active phase: Phase 8 — Maintenance / Service Requests
PR: Phase 8 PR 8.1 — Maintenance audit and implementation plan

## Prerequisite confirmation

Phase 7 is complete and the Phase 7 completion report is present on this branch at `docs/PHASE_7_OWNER_MODEL_COMPLETION_REPORT_2026_05_18.md`. That report explicitly keeps owner accounting, owner statements, and ledger behavior deferred and recommends starting Phase 8 only after the Phase 7 PR is merged.

## Executive conclusion

The safest Phase 8 path is to keep Maintenance / Service Requests as an operational workflow only.

For Phase 8 PR 8.2, improve the existing Supabase-backed Maintenance page in small, reviewable steps:

1. Display the linked property and unit names/numbers in the list.
2. Keep filtering by property and status, and optionally add simple text/priority filters if they only filter already-loaded request rows or are added through the existing service boundary.
3. Keep create and status transitions simple.
4. Add an auditable, non-financial update path for status/resolution metadata if needed.
5. Add service/hook/page tests around the existing lifecycle.

Do **not** automatically create expenses, invoices, payments, receipts, owner chargebacks, vendor payments, or ledger/accounting entries from maintenance requests in Phase 8.

## Current maintenance model inventory

### Database table

The current schema has one maintenance-specific table: `public.maintenance_requests`.

Columns currently defined:

| Column | Current meaning | Notes |
| --- | --- | --- |
| `id` | Request primary key | `uuid`, generated by database. |
| `property_id` | Required property reference | Foreign key to `public.properties(id)` with `on delete restrict`. |
| `unit_id` | Optional unit reference | Foreign key to `public.units(id)` with `on delete restrict`. |
| `title` | Required request title | User-facing summary. |
| `description` | Optional details | Free text. |
| `priority` | Required priority | Check-constrained to `low`, `medium`, `high`, `urgent`. |
| `status` | Required workflow status | Check-constrained to `open`, `in_progress`, `resolved`, `closed`. |
| `assigned_to` | Optional assignee text | Plain text only; not a person/vendor/employee foreign key. |
| `cost` | Numeric cost field | Required, defaults to `0`, check-constrained to non-negative. It is not currently wired to expenses or accounting. |
| `resolved_at` | Optional resolution timestamp | Set by the service when status becomes `resolved` or `closed`; cleared for other statuses. |
| `created_at` | Created timestamp | Database default. |
| `updated_at` | Updated timestamp | Maintained by `public.set_updated_at()`. |
| `deleted_at` | Soft-delete/archive marker | Queried as `null` by the current service. |

### Indexes, trigger, and RLS

The maintenance migration adds:

- Partial index on `property_id` for non-deleted maintenance rows.
- Partial index on `status` for non-deleted maintenance rows.
- `maintenance_requests_set_updated_at` trigger using `public.set_updated_at()`.
- Row-level security enabled on `maintenance_requests`.
- A broad authenticated-user management policy in the initial maintenance migration.
- Later RLS hardening includes `maintenance_requests` in the set of tables where RLS is forced.

### Generated/handwritten TypeScript type coverage

`artifacts/rentrix/src/types/database.ts` includes a `maintenance_requests` table type with the same columns and unions:

- `priority: 'low' | 'medium' | 'high' | 'urgent'`
- `status: 'open' | 'in_progress' | 'resolved' | 'closed'`
- `Insert` requires `property_id`, `title`, `priority`, `status`, and `cost`.
- `Update` is a partial row shape.

`artifacts/rentrix/src/types/domain.ts` currently exports common domain aliases for properties, contracts, invoices, expenses, payments, people, units, owners, and property ownership, but does **not** export a `Maintenance` alias. The maintenance service defines its own `Maintenance` type from `Database['public']['Tables']['maintenance_requests']['Row']`.

## Current UI/service usage

### Route and page

The route file `artifacts/rentrix/src/routes/_protected.maintenance.tsx` exports `MaintenancePage` from `@/features/maintenance/maintenance-page`, so the current `/maintenance` route is wired to the active Maintenance feature.

The active page:

- Uses React local state only for UI filters (`statusFilter`, `propertyFilterId`) and form state.
- Uses `useMaintenance(statusFilter, propertyFilterId)` for maintenance rows.
- Uses `useProperties({ search: '', status: 'all', page: 1, pageSize: 200 })` to populate property filter/select options.
- Uses `useUnits(formPropertyId)` to populate unit choices for the selected property in the create form.
- Creates requests with `property_id`, optional `unit_id`, title, description, priority, `status: 'open'`, `assigned_to: null`, `cost: 0`, and `resolved_at: null`.
- Displays only request title, status, priority, and next status action in the table.
- Does not currently display the linked property or unit in the table, even though those IDs are present.

### Service and hooks

The maintenance service is Supabase-backed:

- `listMaintenance(status, propertyId)` queries `maintenance_requests`, excludes soft-deleted rows, orders by `created_at` descending, filters by status when not `all`, and filters by `property_id` when provided.
- `createMaintenance(payload)` inserts into `maintenance_requests` and returns the inserted row.
- `updateMaintenanceStatus(requestId, status)` updates the status for a non-deleted row and sets `resolved_at` when the new status is `resolved` or `closed`; otherwise it clears `resolved_at`.

The hook layer:

- Defines `maintenanceKeys` as `['maintenance', status, propertyId]` for lists.
- Uses React Query for the list and mutations.
- Invalidates all maintenance queries after create and status update.
- Shows Arabic success/error toasts.

There is also a camelCase `useMaintenance.ts` re-export alias. The code inventory marks it as an orphan alias to consolidate later; this PR should not delete it.

## Current schema/type observations

### Statuses, priorities, and categories

Current maintenance statuses:

- `open`
- `in_progress`
- `resolved`
- `closed`

Current maintenance priorities:

- `low`
- `medium`
- `high`
- `urgent`

Current maintenance categories:

- None. There is no `category`, `request_type`, `trade`, or `vendor_type` column on `maintenance_requests`.
- The unrelated `expenses.category` field exists in the financial expenses table, but maintenance requests are not linked to expenses and should not borrow expense categories without a separate design.

### Create/update/status-change/delete/archive coverage

| Capability | Current state | Notes |
| --- | --- | --- |
| List | Implemented | Supabase-backed list with status and property filters. |
| Create | Implemented | UI creates open requests with `cost: 0`. |
| Status change | Implemented | UI supports `open → in_progress → resolved → closed`. |
| Full edit/update | Not implemented | No UI/service method for title, description, property, unit, priority, assignee, or cost edits. |
| Delete/archive | Schema-supported but not implemented in maintenance service/UI | `deleted_at` exists and list/status queries exclude deleted rows, but there is no maintenance soft-delete/archive function or button. |
| Restore | Not implemented | No restore workflow. |
| History/audit trail | Not implemented | Only `updated_at` and `resolved_at`; no status history table. |

### Financial coupling

There is no current maintenance financial posting behavior:

- The maintenance table has a `cost` field, but the create form always sends `cost: 0`.
- The maintenance service never inserts or updates `expenses`, `invoices`, `payments`, `receipts`, or accounting/ledger tables.
- The expenses table is property-scoped and independent from maintenance records.
- Financial reports use `expenses`, `payments`, and invoices, not maintenance request rows.

Therefore, a maintenance request should be treated as operational data unless and until a later phase explicitly designs financial integration.

## Data relationship map

Current implemented relationships:

```text
properties.id
  ├─ units.property_id
  ├─ contracts.property_id
  ├─ expenses.property_id
  ├─ property_owners.property_id
  └─ maintenance_requests.property_id

units.id
  ├─ contracts.unit_id
  └─ maintenance_requests.unit_id

people.id
  └─ contracts.tenant_id

owners.id
  └─ property_owners.owner_id
```

Maintenance currently links to:

- Property: yes, required by database and service/UI.
- Unit: yes, optional by database and create UI.

Maintenance does **not** currently link to:

- Tenants/people.
- Normalized owners.
- Contracts.
- Expenses.
- Invoices.
- Payments.
- Receipts.
- Ledger/accounting entries.
- Vendors.
- Communication logs or WhatsApp/SMS messages.

The nearest operational inference available today is indirect: a maintenance request has a property and optionally a unit; contracts can also reference property/unit/tenant, but there is no enforced maintenance-to-contract or maintenance-to-tenant link.

## Risks

1. **Financial side effects are high risk.** Automatically converting maintenance costs into expenses, owner chargebacks, tenant invoices, vendor payments, or ledger entries would change financial calculations and reporting.
2. **The `cost` field can be misleading.** It exists in schema but is currently always created as `0` by the UI and has no posting semantics.
3. **No category model exists.** Adding categories in the UI without a schema/design could create local-only state or overload financial expense categories.
4. **No assignee/vendor model exists.** `assigned_to` is text and should not be treated as a vendor/payment entity.
5. **No audit history table exists.** Status changes are auditable only through row timestamps, not a complete event log.
6. **Current table display hides relationships.** Users can create property/unit-linked requests but cannot see those relationship labels in the list.
7. **Alias hook file remains ambiguous.** `useMaintenance.ts` is a re-export alias marked orphan by the inventory, but deleting it should be a separate cleanup PR.
8. **RLS is broad.** Current authenticated management policy is acceptable for existing app shape, but more granular maintenance permissions should be designed separately, not bundled with UI cleanup.

## Recommended Phase 8 implementation path

### PR 8.1 — Audit and plan

This PR should remain docs-first and make no runtime behavior changes.

### PR 8.2 — Maintenance operational UI improvements

Recommended small scope:

1. Keep Maintenance operational-only.
2. Keep Supabase service/hook boundaries; do not query Supabase directly from presentational UI.
3. Improve list display by showing property title and unit number, using the existing `useProperties` and `useUnits`/unit service paths or a narrowly typed service projection.

- Permission/RLS review if roles are introduced.
